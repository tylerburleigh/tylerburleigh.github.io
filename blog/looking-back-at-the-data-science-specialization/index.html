Looking back at the Data Science Specialization
July 29, 2019
[ Upskilling ]
data science R Coursera specialization experience

I’ve just completed the Data Science Specialization on Coursera, earning my certificate.

I decided to take this specialization because I’d just become a data scientist (technically a “Quantitative Researcher”), but I didn’t know what “data science” was. Was it just applied statistics repackaged for industry? At the very least, I figured I’d learn the sorts of terms that data scientists used by convention to refer to things, and I’d be better able to communicate with other data scientists. After all, this specialization is one of the most popular online data science MOOCs. I also wanted to check my knowledge of R because, although I felt I had a solid grasp already, I hadn’t received any formal training and I’d never had an R peer to compare notes with. In short: I was confronting my impostor syndrome head-on.

On the specialization “about” page, it says it’s appropriate for individuals with “beginner level experience” in programming (it actually says “in Python” but I’m pretty sure that’s an error) and recommends “familiarity with regression”. Going into the course, I’d say I was advanced in both of these areas. I’d completed a PhD in psychology which involves a lot of inferential statistics, including regression, and on top of that I’d coded all of my thesis analyses in R. I’d also worked as a freelance software developer for a few years, mostlyworking in JavaScript/PHP, but hey, there’s a lot of knowledge transfer between programming languages.

In the list of courses, there were also 3 that I thought I could really learn a lot from:

  • Practical Machine Learning
  • Developing Data Products
  • Data Science Capstone

I wanted an introduction to machine learning in a programming language that I already knew pretty well. I’d recently done the Kaggle Titanic machine learning challenge. I wanted to know more about machine learning, and I wanted to learn in a more structured environment so I could be sure I was getting off on the right foot.

I also wanted to learn more how to use R to build software. I knew that I enjoyed building software, but most of the tools I’d created were coded in JavaScript and I didn’t see a lot of potential to use JavaScript to create the sort of tools that I imagined would be useful in the world of data science. I imagined these tools would have to be more “statistically literate” and better able to process large amounts of data. My experience with R had been mostly statistical programming and so I was excited to learn what it could do for more general-purpose applications.

I really did learn a lot in these three courses. I learned how to use Shiny to build R prototypes and web-apps, how to write R packages, I learned about decision trees, k-means clustering, natural language processing (NLP), and about the apply family of functions. I was so inspired by the Data Products course that I started to author my own package (pyMTurkR). In the capstone I faced a situation where the data was truly in the realm of “big data”. The data tables were so large that they couldn’t fit into the memory on my machine, and so I was forced to learn two ways data scientists deal with that problem – stream processing via Spark / sparklyr and spinning up EC2 cloud servers that just have gobs and gobs of RAM.

Looking back at this specialization, and in parallel looking at the last 6 months of actually working in the role of a data scientist, I’ve come to the conclusion that a “data scientist” is someone who never stops learning new ways to solve problems (involving data). And one of the best ways to “grease the wheels” of creative problem solving is to experiment with new concepts and frameworks and techniques.
